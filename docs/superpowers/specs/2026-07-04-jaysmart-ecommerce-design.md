# Jaysmart — Preowned Laptop Ecommerce (Design Spec)

**Status:** Draft (awaiting user review)
**Date:** 2026-07-04
**Owner:** Ayomikun Ayobami

## 1. Overview

Jaysmart is an SEO-friendly ecommerce site for certified preowned laptops in Nigeria. It offers two purchase paths — full online checkout via Paystack, and WhatsApp inquiry for buyers who prefer chat. Content, inventory, orders, and customers are managed through an embedded Payload CMS admin. The site is built to rank well on Google (Core Web Vitals, structured data, dynamic sitemap) and to convert visitors from either payment path.

**Success criteria for v1:**

- Every published laptop is a Google-indexable page with Product structured data (price, availability, brand).
- Customer can complete a purchase online end-to-end with Paystack in test mode.
- Admin can add a new laptop, upload photos, publish, and see it live on the storefront within one minute.
- Homepage LCP under 2.5s on a throttled 4G connection.
- Both admin and customer authentication flows work with password reset.

## 2. Tech stack

| Concern | Choice |
|---|---|
| Framework | Next.js 15 (App Router, SSR + SSG) |
| UI library | MUI v6 with App Router integration (`AppRouterCacheProvider`) |
| CMS / backend | Payload CMS 3 (embedded in the Next.js app) |
| Database | Postgres (Railway managed) |
| Media storage | Railway object storage (S3-compatible) via Payload's S3 storage plugin |
| Payments | Paystack (inline modal + webhook) |
| Email | Resend + React Email templates |
| Client state | Zustand (cart, wishlist) |
| Validation | Zod |
| Testing | Vitest (unit + integration), Playwright (E2E) |
| Errors | Sentry |
| Analytics | PostHog |
| Logging | Pino |
| Hosting | Railway (app + Postgres + object storage in one project) |

Rationale for key choices:

- **Payload embedded, not standalone.** For the scale of ~300 laptops and a small team, one deploy is simpler and faster. SSR pages read Payload data via the in-process local API — no HTTP hop, no CORS.
- **Railway everything.** Single dashboard, single bill, single failure domain to reason about. Trade-off: not the cheapest at scale, but simplest to run.
- **Prices in kobo (integer minor units).** Avoids float rounding and matches Paystack's API contract.

**Data flow (read path):** server component → `getPayload({ config })` → Payload local API → Postgres. No network hop. Types are auto-generated by Payload and imported directly.

**Data flow (write path — checkout):** client cart → server action → Payload local API creates `Order` (pending) → Paystack modal → webhook → Payload updates `Order` to paid + decrements `Laptop.stock` in a transaction → email dispatched.

## 4. Data model

All Payload collections. Field lists below are the source of truth for the implementation plan.

|---|---|---|
| `title` | text, required | e.g. "HP EliteBook 840 G5 Core i5 8th Gen" |
| `slug` | text, unique, auto | Generated from title on save |
| `brand` | relationship → Categories | Filter: `type=brand` |
| `category` | relationship → Categories | Filter: `type=useCase` |

| `specs.ram` | number | GB |
| `specs.storage` | text | e.g. "512GB SSD" |
| `specs.screenSize` | number | inches |
| `specs.batteryHealth` | number | percent, 0–100 |
| `specs.os` | text | |
| `gallery` | array of Media relationships | First image is hero |
| `description` | rich text | |
| `warrantyDays` | number, default 7 | |

| `status` | select | `draft` \| `published` \| `sold` |

### 4.5 `Users` (admin users — separate auth)

| Field | Type | Notes |
|---|---|---|
| `email` | email, unique, required | |

## 5. Pages & SEO

### 5.1 Page inventory

| Route | Rendering | Auth | Indexable |
|---|---|---|---|
| `/` | SSG, revalidate 5 min | — | Yes |
| `/laptops` | SSR | — | Yes (canonical on filtered variants) |
| `/laptops/[slug]` | `generateStaticParams` at build + dynamic on-demand for new slugs; `revalidatePath` on publish | — | Yes |
| `/compare` | SSR | — | No (`noindex`) |
| `/cart` | Client-only | — | No |
| `/checkout` | Client + server action | — | No |
| `/order/[id]` | Server | Order owner or signed link | No |
| `/account/login`, `/account/register` | Client | — | No |
| `/account/**` | Server | Customer | No |
| `/admin/**` | Payload | Admin user | No, blocked in robots.txt |

### 5.2 SEO plan

**Metadata:** `generateMetadata()` on every public route. Product pages read `seo.*` from Payload with fallbacks.

**Structured data (JSON-LD, in a shared helper `lib/seo.ts`):**
- Product schema on `/laptops/[slug]` — `name`, `image`, `description`, `sku`, `brand`, `offers.price`, `offers.priceCurrency: NGN`, `offers.availability`, `itemCondition`.
- BreadcrumbList schema on `/laptops` and `/laptops/[slug]`.
- Organization + LocalBusiness schema on homepage, pulled from the Settings global.

**Sitemap:** `app/sitemap.ts` returns a `MetadataRoute.Sitemap` — homepage, `/laptops`, each published laptop, each category. Cached for 1 hour.

**Robots:** `app/robots.ts` — allow public paths, disallow `/admin`, `/api`, `/checkout`, `/cart`, `/account`, `/order`.

**Canonical URLs:** every page emits an absolute canonical. `/laptops?brand=hp&useCase=programming` canonicals to `/laptops?brand=hp&useCase=programming` (stable ordering); homepage-equivalent filter combinations canonical to `/laptops`.

**Locale:** `<html lang="en-NG">`. Prices formatted with `Intl.NumberFormat('en-NG', { style: 'currency', currency: 'NGN', maximumFractionDigits: 0 })`.

### 5.3 Performance / Core Web Vitals

- `next/image` for all images; `sizes` attribute set per layout; hero image on homepage and product detail is `priority`.
- `next/font` for Inter, self-hosted, `display: swap`.
- MUI v6 SSR via `AppRouterCacheProvider` from `@mui/material-nextjs/v15-appRouter` — prevents FOUC and hydration mismatch.
- Bundle budget: homepage JS ≤ 150 KB gzipped; product page ≤ 200 KB.
- Zustand for cart/wishlist (~1 KB) instead of Redux.
- Streaming SSR where useful (listing page can stream filter panel + product grid separately).

## 6. Checkout & payments

### 6.1 Online path (Paystack)

1. Cart persists in Zustand + localStorage. Fully client-side until checkout.
2. `/checkout` renders a form (Zod-validated) capturing delivery address and (for guests) email/name/phone.
3. Submit invokes a **server action** that:
   - Refetches each laptop from Payload and verifies `stock >= 1`.
   - Computes `subtotal`, `deliveryFee` (Lagos vs. other based on `deliveryAddress.state`), `total`.
   - Creates an `Order` with `paymentStatus: pending`, snapshots each laptop.
   - Returns `{ orderId, amountKobo, paystackReference }`.
   - **Note:** stock is NOT reserved at this point. Two customers can both create pending orders for the same unit. Only the first webhook to complete wins (see section 6.3); the losing customer is auto-refunded.
4. Client opens Paystack inline modal with the returned reference.
5. On success, client redirects to `/order/[id]?ref=xxx`.
6. **Webhook** (`/api/paystack/webhook`) is the source of truth:
   - Verifies HMAC-SHA512 signature with `PAYSTACK_WEBHOOK_SECRET`.
   - Looks up order by `paymentReference`; if already `paid`, returns 200 immediately (idempotent).
   - In a Postgres transaction: sets order to `paid`, decrements `stock` on each item (with `SELECT ... FOR UPDATE` on the laptop rows).
   - Sends receipt email to customer and notification email to admin.

### 6.2 WhatsApp path

- Every "chat" or "WhatsApp us" button constructs a `wa.me/<number>?text=<encoded>` URL through `lib/whatsapp.ts`.
- Templates are context-aware (product card, product detail, hero, floating FAB).
- Every click fires a non-blocking `POST /api/whatsapp-inquiry` that creates a `WhatsAppInquiries` record.

### 6.3 Failure modes

| Failure | Handling |
|---|---|
| Callback fails but payment succeeds | Webhook still marks order paid; receipt email carries order link |
| Webhook signature invalid | Return 401; log to Sentry |
| Paystack API down at checkout | Server action returns error; UI shows fallback "Try WhatsApp" with pre-filled link |
| Out-of-stock at checkout | Server action returns per-item error; cart drawer highlights sold items |
| Duplicate webhook | Idempotent by `paymentReference` check |
| Concurrent purchases of the same unit | Transaction + `FOR UPDATE` ensures only one wins; loser gets a refund flow (manual v1) |

### 6.4 Delivery fee

Two-tier flat rate configured in Settings global:
- Lagos state → `deliveryFeeLagos`
- All other states → `deliveryFeeOther`

Computed in the server action, not the client, so client can't tamper.

## 7. Admin & customer accounts

### 7.1 Admin at `/admin`

Provided by Payload. Two roles (`admin`, `staff`) with per-collection access rules defined in each collection's `access` config. Bootstrap admin created on first deploy from `ADMIN_EMAIL`/`ADMIN_PASSWORD` env vars.

### 7.2 Customer at `/account/*`

Payload-authenticated `Customers` collection (separate from `Users`). Cookie-based auth (`httpOnly`, `secure`, `sameSite=lax`). Password reset via Resend email with signed short-lived tokens (Payload built-in). Rate limit: 5 login attempts per IP per 5 min.

Routes:
- `/account/login`, `/account/register`, `/account/forgot-password`
- `/account` — dashboard
- `/account/orders`, `/account/orders/[id]`
- `/account/wishlist`
- `/account/settings`

Wishlist merge: on login, any `localStorage` guest wishlist is merged into the account's `wishlist` array.

Guest checkout is fully supported — an account is never required to buy.

## 10. Explicit non-goals for v1

- Blog / buying guides (structure allows adding later — new Payload collection + route)
- Coupon / discount codes
- Multi-currency
- Social login (Google/Apple)
- SMS 2FA
- Location-based delivery pricing beyond two tiers
- Partial refunds automated in-app (handled manually via Paystack dashboard)
- A/B testing framework (PostHog can add later without code changes)
- Redis session store

## 11. Design system

MUI theme built from the Material 3 tokens in the provided wireframe. Palette (primary `#003178`, secondary `#1b6d24`, error `#ba1a1a`, surface `#f7f9ff`, etc.), typography (Inter with `display-lg`, `headline-md`, `body-lg`, `price-display`, `label-bold` scales), border radii (`sm 2px / md 4px / lg 8px / xl 12px`) all set once in `lib/theme.ts`. Material Symbols Outlined loaded via `next/font` for icon parity with the wireframe.
